### Added
- Initial repository scaffold: `pyproject.toml`, `src/` layout, MIT license, CI workflow, pre-commit hooks.
- `swatplus-ai version` CLI command as a first end-to-end smoke check.
- Parser slice 1 — control & inventory: `time.sim`, `print.prt`, `file.cio`, `codes.bsn`, `parameters.bsn`, `object.cnt`.
- Parser slice 2 — HRU core: `hydrology.hyd`, `hru-data.hru`, `landuse.lum`, `plant.ini`, `soils.sol`, `topography.hyd`, `nutrients.sol`.
- Parser slice 3 — weather: `weather-sta.cli`, `weather-wgn.cli`, generic `pcp/tmp/slr/hmd/wnd.cli`.
- Parser slice 4 — management & lookups: `management.sch`, `fertilizer.frt`, `tillage.til`, `pesticide.pes`, `harv/graze/irr/fire/sweep/chem_app.ops`, `cntable.lum`, `cons_practice.lum`, `ovn_table.lum`.
- Parser slice 5 — connectivity / topology: `hru.con`, `aquifer.con`, `chandeg.con`, `reservoir.con`, `rout_unit.con`, `ls_unit.def`/`.ele`, `rout_unit.def`/`.ele`/`.rtu`, `aqu_catunit.ele`.
- Parser slice 6 — routing bodies: `aquifer.aqu`, `initial.aqu`, `channel-lte.cha`, `hyd-sed-lte.cha`, `nutrients.cha`, `initial.cha`, `reservoir.res`, `hydrology.res`, `nutrients.res`, `sediment.res`, `initial.res`, `wetland.wet`, `hydrology.wet`. Shared `parse_initial_any` backs the three `initial.{aqu,cha,res}` files.
- Parser slice 7 (partial) — HRU initial / chemistry: `soil_plant.ini`, `om_water.ini`.
- Parser slice 8 (partial) — calibration / change: `cal_parms.cal`, `codes.sft`, `wb_parms.sft`, `water_balance.sft`, `plant_gro.sft`, `plant_parms.sft`.
- Parser slice 9 — decision tables: `lum.dtl`, `res_rel.dtl`, backed by a shared `_decision_table.py` helper. Table count on line 2 is informational; tables are read until EOF so hand-edited files parse correctly.
- `TxtInOutProject.read()` orchestrator covering slices 1–9; every optional file is tolerated when absent.
- Output parsers slice A — annual-average / yearly summaries as pandas DataFrames: `basin_wb_aa`, `basin_pw_aa`, `basin_ls_aa`, `basin_nb_aa`, `hru_wb_aa`, `hru_pw_aa`, `hru_ls_aa`, `hru_nb_aa`, `channel_sd_aa`, `channel_sdmorph_aa`, `aquifer_aa`, `reservoir_aa`, `wetland_aa`. Shared `read_aa_output` reader handles title/header/units preambles, multi-word text columns (`plant_cov`), duplicate header names (`null` separators, repeated `deg_btm`/`deg_bank`), and short rows. Wired into `TxtInOutProject.outputs`; every DataFrame is optional so un-run projects still parse.
- `TxtInOutProject.topology.outfall_channels()` returns names of channels with `out_tot == 0` from the already-parsed `chandeg.con`.
- New runtime dependencies: `pandas>=2.2`, `pyarrow>=16`.
- LLM gateway (Phase 1 Step 5):
  - `swatplus_ai.llm.interface` — `Message`, `LLMResponse`, `LLMBackend` Protocol, `LLMError` / `AuthError` / `RateLimitError`.
  - API-key backends for Anthropic (`x-api-key`) and OpenAI (`Bearer`) speaking raw HTTP via `httpx` (no vendor SDKs), with streaming SSE support.
  - `MockBackend` for deterministic tests.
  - Token storage: in-memory default plus an optional OS-keyring store (install with the new `secrets` extra: `pip install 'swatplus-ai[secrets]'`).
  - **Experimental, opt-in** OAuth passthrough backends for Anthropic and OpenAI using the Authorization Code + PKCE flow that Claude Code and OpenAI Codex CLI use. Subscription-bound; disabled by default until the upstream client IDs are pinned.
- Diagnostic rule engine infrastructure (Phase 1 Step 4 slice 4.1): `swatplus_ai.diagnostics` exposes `Finding`, `Rule`, `CheckResult`, `DiagnosticEngine`, and the `register_check` decorator. YAML-loaded rules reference Python check functions by name via a registry; `DiagnosticEngine.from_builtin_rules()` loads every `*.yaml` shipped under `swatplus_ai/diagnostics/rules/` and fails loudly at construction time if a rule references an unregistered check or duplicates an id across files. `.run(project, stage=...)` filters rules by lifecycle stage and skips rules whose `requires` attrs are absent on the project. No rules bundled yet — ships in slices 4.2 and 4.3.
- Diagnostic rules slice 4.2 — five pre-run `setup.*` rules bundled with `DiagnosticEngine.from_builtin_rules()`: `setup.files_present`, `setup.sim_period_sanity`, `setup.warmup_ratio`, `setup.mgt_date_order`, `setup.object_count_consistency`. Covers the structural input-side problems (missing control files, inverted / zero-length / too-short simulation windows, implausible warm-up ratios, out-of-order management schedules, `object.cnt` vs `.con` count mismatches) that SWAT+ either crashes on or silently runs through. `CheckResult` gains an optional `severity` override so a single rule can emit findings at different severities (used by `setup.sim_period_sanity`). Check functions live under `swatplus_ai/diagnostics/checks/` and are auto-registered on package import.
- Diagnostic rules slice 4.3 — five new rules close the Module-1 (pre-run + first-pass evaluation) set, bringing `DiagnosticEngine.from_builtin_rules()` to ten rule ids: `hru.fk_consistency` (setup, error) guards the four FKs on every `hru-data.hru` row; `chan.routing_topology` (setup, error/warning) covers outfall existence, `out_tot` consistency, `frac` sums, downstream range, and cycle detection in `chandeg.con`; `setup.pet_method_vs_climate` (setup, warning) branches on `codes.bsn.pet` against `weather-sta.cli` sources; `wx.source_consistency` (setup, error) cross-checks `weather-sta.cli` against the per-variable `.cli` indices and their on-disk files; `wb.et_precip_ratio` (evaluation, warning) flags implausible basin-scale ET/precipitation ratios on `basin_wb_aa`. `DiagnosticEngine` now resolves dotted `requires` paths (`outputs.basin_wb_aa`) attribute-by-attribute.
- Telemetry core (Phase 1 Step 4.5 slice 4.5.1): new `swatplus_ai.telemetry` module with a frozen `Event` model covering an eight-type event taxonomy (`session_start`, `session_end`, `file_parsed`, `parse_error`, `rule_evaluated`, `finding_emitted`, `llm_call`, `llm_tool_call_chosen`, `user_action`), three sinks (`JsonlFileSink` append-only JSON Lines, `InMemorySink` for tests, `NullSink` for disabled / post-failure state), and a pure `redact()` utility that scrubs absolute paths to basenames and Anthropic/OpenAI keys, emails, and long opaque tokens to `<REDACTED>` — exercised by an adversarial test bank. Process-wide `emit(event_type, **fields)` entrypoint is non-blocking (swallows sink exceptions, warns once, switches the session to `NullSink`) and buffers up to 50 events before the first `configure()` call so early `session_start` events aren't lost. `swatplus-ai telemetry status|enable|disable` manages a global flag at `~/.swatplus-ai/config.toml` (or `$XDG_CONFIG_HOME/swatplus-ai/config.toml`); the `SWATPLUS_AI_NO_LOG=1` env var overrides the config. `.swatplus-ai/` added to `.gitignore` so per-project log dirs never enter git. No instrumentation of parser / engine / LLM gateway in this slice — that is 4.5.2.
- Telemetry instrumentation (Phase 1 Step 4.5 slice 4.5.2): `TxtInOutProject.read` now emits `file_parsed` (fields: `filename`, `rows`, `duration_ms`) for every required and optional input file it successfully parses, and `parse_error` (fields: `filename`, `exception`, `line` for `ParseError`) immediately before re-raising. Absent optional files stay silent — the log doesn't fill with misses on projects that legitimately ship a trimmed file set. `OutputsNamespace.read` instruments the 13 annual-average output files the same way, with `rows = len(df)`. `DiagnosticEngine.run` emits `rule_evaluated` (fields: `rule_id`, `stage`, `finding_count`, `duration_ms`) once per rule that matches the run's stage, preceded by one `finding_emitted` event (fields: `rule_id`, `severity`) per finding the rule produced — so log readers see N finding events followed by the rule's summary. Rules whose `requires` attrs are absent on the project emit a single `rule_evaluated` with `skipped=True, finding_count=0, duration_ms=0`; rules whose stage doesn't match the run are silent. No behavioral change to either module — findings are returned identically whether telemetry is on or off, and the existing 415-test suite passes unchanged. The `redact()` token regex now additionally requires a digit inside the candidate run so long symbolic identifiers (rule ids like `setup.object_count_consistency`, Python class names) pass through untouched; real API keys, SHA hashes, and base64 tokens always contain digits.
- Log inspection CLI (Phase 1 Step 4.5 slice 4.5.3): read-side `swatplus-ai logs` sub-app with three commands — `logs list` indexes every session log under `<cwd>/.swatplus-ai/logs/` (Rich table: 8-char session id, ISO `started`, event count, filename), `logs show [--last N] [--session UUID_PREFIX]` pretty-prints one session with an absolute `HH:MM:SS.mmm` UTC timestamp plus an event-type-aware one-line summary (filename+rows+duration for `file_parsed`, rule_id+finding_count+duration for `rule_evaluated`, etc.), and `logs export --output PATH [--session UUID_PREFIX]` does a byte-for-byte `shutil.copy` of a session's JSONL (refusing to overwrite an existing path). Supporting pure-helper module `swatplus_ai.telemetry.logs` exposes `default_logs_dir()`, `find_sessions()`, `load_session()`, `resolve_session()`, `session_id_from_path()`, and a frozen `SessionSummary` dataclass. `load_session` tolerates mid-stream corruption (blank / unparseable-JSON / failed-`Event`-validation lines are dropped) so a disk-full crash mid-session doesn't block log reads. `resolve_session` requires a ≥4-char uuid prefix and raises `ValueError` on too-short / ambiguous / unmatched hints, `FileNotFoundError` on a missing directory or empty logs dir. Empty session files still appear in `find_sessions` — a sink that crashed before writing is the exact failure telemetry exists to record. Root CLI gains `app.add_typer(logs_app, name="logs")`. New README "Session logs" section covers location, JSONL format, redaction guarantees, the three commands, and how to disable (`telemetry disable` or `SWATPLUS_AI_NO_LOG=1`). No production `JsonlFileSink` wiring in this slice — that arrives with `swatplus-ai check <path>` in Step 7; until then these commands already work against any session file the test suite or a hand-wired sink produces.
- Prompt assembler — Module 1 slice 6.1 (Phase 1 Step 6 part 1/2): new `swatplus_ai.prompts` package exposing `build_module1_prompt(findings, project_summary, static_passages=None) -> list[Message]`, the frozen `ProjectSummary` pydantic model (with a `ProjectSummary.from_project(project)` factory extracting `sim_start_year`, `sim_end_year`, `warmup_years`, `n_hrus`, `n_channels`, `n_aquifers`, `n_subbasins`, `pet_method`, `basin_area_km2` from a `TxtInOutProject` and gracefully degrading every `object.cnt`-sourced count to `None` when the file is absent), and a frozen `StaticPassage` contract (`id`, `title`, `body`, `source`) that the Phase 2 retrieval layer will populate. The system prompt body lives as versioned markdown data in `src/swatplus_ai/prompts/module1_system.md` with three `{{VAR}}` placeholders (`{{PROJECT_SUMMARY}}`, `{{FINDINGS}}`, `{{STATIC_PASSAGES}}`) rendered via plain string substitution; `{{VAR}}` over `{VAR}` was chosen so `str.format` / f-string braces elsewhere in the template never collide with substitution, and over Jinja to keep the slice dependency-free. Findings render in error → warning → info order, alphabetised by id within each bucket; empty `references` render as `—` rather than a blank label. `static_passages=None` and `static_passages=[]` both render the literal `"None provided."` — deliberate since the retrieval layer isn't wired yet and a silent empty section would let the LLM think passages were truncated. Template is loaded via `importlib.resources.files("swatplus_ai.prompts").joinpath("module1_system.md").read_text()` so the lookup works identically from a wheel install and a source checkout; the `.md` file ships in the wheel via a new `tool.hatch.build.targets.wheel.force-include` entry. Returns `list[Message]` (not a single string) so the caller can hand the list straight to any `LLMBackend.complete(...)` without the gateway having to re-split a system prefix. 14 new tests under `tests/prompts/` cover message shape, placeholder substitution, severity + id ordering, `None` / `[]` / populated passage rendering, `from_project` field extraction on the committed minimal fixture, the `object_cnt=None` degrade path, a literal "never invent" contract pin on the template body, a "diagnostic" / "report" pin on the opening user message, and the empty-references dash rendering. No mutation of `Finding`, `Message`, `TxtInOutProject`, or `DiagnosticEngine`. No retrieval / RAG, no response formatter (slice 6.2), no telemetry emission (slice 7) — deliberately deferred.
- LLM gateway telemetry (Phase 1 Step 5 closure): every `_BaseBackend.complete()` call (`AnthropicBackend`, `OpenAIBackend`, plus any future OAuth-flavoured subclass that inherits from `_AnthropicProtocol` / `_OpenAIProtocol`) and every `MockBackend.complete()` call now emits an `llm_call` event via the slice-4.5.1 `telemetry.emit()` surface. Fields: `provider` (`"anthropic"` / `"openai"` / `"mock"`, sourced from a new `PROVIDER: ClassVar[str]` on each protocol class so api-key and OAuth variants report the same tag), `model`, `input_tokens`, `output_tokens`, `finish_reason` (reuses the normalized `LLMResponse.finish_reason` on the happy path; `"error"` on the error path), `duration_ms` (wall-clock via `time.perf_counter`), `streaming=False`, and — only on the error path — `exception` (exception class name). Error emission happens **before** the re-raise so the event lands on disk even if the process dies inside the caller's `except` block (same pattern parser `parse_error` uses). Emission goes through the existing redactor and is a no-op when `SWATPLUS_AI_NO_LOG=1` or the user has run `swatplus-ai telemetry disable`. Zero change to the public contract of `LLMBackend` / `LLMResponse` / `Message` — `contract_identity_across_telemetry_toggle` test pins bit-identical responses on and off. Known limitations deliberately deferred: (1) `stream()` does **not** emit — streaming token-accounting requires provider-specific usage-frame logic (OpenAI's `stream_options.include_usage`, Anthropic's `message_delta.usage`) plus mid-stream error handling, which is its own design exercise; a regression test (`test_stream_does_not_emit`) pins the current behaviour so re-enabling later is a conscious edit. (2) `llm_tool_call_chosen` is **not** implemented — the gateway has no tool-use parsing yet (no `parse_response` path extracts `tool_use` blocks), so stubbing this event would be a lie on disk until Module-1 tools land.
